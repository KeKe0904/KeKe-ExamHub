# 学习工作 · KeKe ExamHub 考试信息管理系统

## 一、Demo 简介

**KeKe ExamHub** 是一个面向学校与考试机构的**考试信息管理系统**，采用前后端分离架构，一套代码同时提供**四端入口**：管理员后台、教室端大屏、学生端、教师端。用 TRAE IDE 从零开始全栈开发，覆盖考试发布、教室分配、教师/学生管理、公告推送、域名 SSL、AI 助手等完整业务闭环。

- **产品形态**：Web 应用（前后端分离，React + Node.js/Fastify + MySQL）
- **面向谁**：学校教务处、考试机构、培训班管理员，以及对应的教师、学生、监考人员
- **部署方式**：一键部署脚本 `install.sh`（Debian/Ubuntu 全自动，8 步完成）

### 核心功能

| 功能 | 说明 |
|------|------|
| **四端协同** | 管理员后台全功能管理 / 教室端大屏监考倒计时 / 学生端查看考试安排 / 教师端查看班级和考试，JWT 四端互斥认证 |
| **考试全流程管理** | 创建考试 → 分配教室 → 分配监考教师 → 学生按班级关联考试 → 教室端大屏实时倒计时监考 |
| **AI 助手（Function Calling）** | 管理员可用自然语言操作后台：27 个工具调用，分 safe / dangerous / forbidden 三级，危险操作需二次确认 |
| **一键部署 + 安装向导** | `install.sh v4.0` 全自动部署（含域名 DNS 校验 + SSL 证书四选项），首次访问 `/setup` 向导建库建表 |
| **安全加固** | JWT 强密钥校验、IP 黑名单、域名访问控制、XSS 过滤、速率限制、审计日志、命令注入防护 |

## 二、Demo 创作思路

### 灵感来源

我是落梦陳（B站/抖音同名），平时会接触到学校考试管理的真实场景。传统考试信息发布靠纸质公告栏 + 群消息通知，存在几个痛点：

1. **信息触达不及时**：考生经常错过考试时间地点变更通知
2. **教室调度混乱**：多场考试同时进行时，教室分配靠 Excel 手动排，容易冲突
3. **监考老师安排不透明**：监考安排只发在教务群，老师经常忘记自己的监考任务
4. **考场信息分散**：学生不知道自己在哪间教室、座位号是多少

市面上虽然有考试系统，但大多要么是重型 SaaS（价格贵、定制难），要么只是简单的信息发布页（功能太弱）。我想做一个**轻量、自部署、四端打通**的系统。

### 想解决的问题

- **管理员**：一个后台搞定考试、教室、教师、学生、公告的全链路管理，不用在多个工具间切换
- **学生**：一个页面看到自己所有考试的时间、地点、座位号，不用翻群消息
- **教师**：清楚知道自己监考哪场考试、在哪个教室，不用问教务
- **考场**：教室一体机打开就是监考大屏，实时倒计时，不用额外装软件

### 为什么做这个方向

1. **真实需求**：这是我自己和身边人真实遇到的痛点，不是凭空想象的
2. **可自部署**：学校对数据安全敏感，不愿意用公有云 SaaS，自部署是刚需
3. **技术挑战适中**：四端认证、实时数据、AI 集成、一键部署，技术栈有深度但不过度
4. **TRAE 适合**：全栈项目，前后端 + 数据库 + 部署脚本，正好发挥 TRAE 全流程开发能力

### 4.3 四端认证系统

JWT 四端互斥认证是最核心的安全设计。用 TRAE 实现了：

- 四套独立的认证中间件（`authMiddleware` / `classroomAuthMiddleware` / `studentAuthMiddleware` / `teacherAuthMiddleware`）
- 每个 token 携带 `role` 字段，中间件校验角色匹配，四端 token 互不能调用对方接口
- 时序攻击防护：账号不存在时仍执行 bcrypt 比较均衡响应时间
- 速率限制：管理员登录 5 次/分钟，学生登录 10 次/分钟

### 4.4 AI 助手（Function Calling + SSE 流式）

这是本项目最有亮点的功能。管理员可以用自然语言操作后台，例如：
- 「帮我查询下周有几场数学考试」
- 「创建一场英语期末考试，7月20日上午9点，在教学楼A101」
- 「把高三1班所有学生导入系统」

用 TRAE 实现的架构：
- 27 个工具，分三级：safe（13 个自动执行）/ dangerous（14 个需二次确认）/ forbidden（AI 拒绝并引导手动操作）
- SSE 流式响应，前端实时显示 AI 回复
- 危险操作弹出红色警告对话框，用户确认后才执行
- 工具执行 30 秒超时 + 结果 4000 字符截断 + 上下文滑窗（保留最近 30 条消息）

### 4.6 安全审计与漏洞修复（v1.2.0 重点）

发布前用 TRAE 对整个项目做了全面安全审计，发现并修复了多个严重漏洞：

| 漏洞 | 严重程度 | 修复方式 |
|------|---------|---------|
| acme.ts 命令注入（execSync 拼接 shell） | 严重 | 改用 execFileSync 参数数组 |
| setup.ts 明文密码写入 .env | 严重 | 安装后移除 ADMIN_PASSWORD |
| acme.ts 路径遍历（domain 参数未校验） | 高 | 增加域名格式校验 |
| auth.ts IP 伪造（信任 X-Forwarded-For） | 高 | 统一使用 getClientIp |
| 前端公告 XSS（dangerouslySetInnerHTML） | 高 | 增加 DOMPurify 二次净化 |
| 分页接口 LIMIT 参数 500 错误 | 高 | pool.execute → pool.query |

### 4.7 开发踩坑记录

开发过程中遇到了不少坑，都在 TRAE 的帮助下逐一解决：

1. **ESM 模块的 `__dirname` 问题**：后端 `package.json` 有 `"type": "module"`，`__dirname` 不可用，用 `import.meta.url` 替代
2. **mysql2 预处理语句 LIMIT 参数 bug**：`pool.execute()` 配合 `LIMIT ? OFFSET ?` 传入 JS 数值会报 `Incorrect arguments to mysqld_stmt_execute`，改为 `pool.query()` 解决
3. **DNS 解析正则 bug**：GNU grep ERE 不支持 `\d`，必须用 `[0-9]`
4. **pm2 spawn 崩溃**：pm2 未安装时 `spawn pm2 ENOENT` 会导致进程崩溃，添加 `.on("error")` 事件监听器
5. **安装向导建表不全**：之前只手动建了 9 张表，缺少 9 张关键业务表，改为加载完整 init.sql

## 六、技术亮点总结

### 6.1 四端 JWT 互斥认证

四端（管理员/教室/学生/教师）各自独立 token，中间件校验 role 字段，互不能越权。81 项安全边界测试全部通过：无 token → 401，跨角色 → 403。

### 6.2 AI 助手分级执行

不是简单的"AI 调接口"，而是完整的工具调用安全体系：
- **safe**：查询类，自动执行（如查询考试列表）
- **dangerous**：写入类，前端弹窗二次确认（如创建考试、导入学生）
- **forbidden**：删除类，AI 直接拒绝并引导手动操作（如删除考试、重装环境）

### 6.3 教室端 IP 信任机制

教室端登录有异常 IP 检测：
- 首次登录 IP 记录为「待审核」
- 同一 IP 登录 ≥ 3 次自动加入可信列表
- 非可信 IP 登录需管理员审核
- 配合 IP 黑名单，防止异地异常登录

### 6.4 一键部署的工业级实现

`install.sh v4.0` 不是简单的脚本，而是包含：
- 错误处理（`set -Eeuo pipefail` + `trap ERR`）
- 日志落盘（`/var/log/examhub-install.log`）
- 断点续传（`--step` 参数）
- DNS 校验（显式查公共 DNS，支持 CDN/NAT 场景）
- SSL 四选项（不配置/上传/检测/自动申请）
