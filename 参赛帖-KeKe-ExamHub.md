# 学习工作 · KeKe ExamHub — 学校考试信息数字化管理平台

> 一套代码，四端协同。让全国数百万间教室的一体机从"高清投影仪"升级为"数字考试终端"。

---

## 一、Demo 简介

**KeKe ExamHub** 是一个面向中小学和考试机构的**全栈考试信息数字化管理平台**，采用前后端分离架构，一套代码同时提供**四端入口**：公众前台（学生/家长浏览）、教室端大屏（一体机监考模式）、教师端、管理员后台。完全使用 TRAE IDE 从零开发，覆盖考试发布、教室分配、监考安排、学生管理、AI 助手、域名 SSL、安全审计等完整业务闭环。

- **产品形态**：Web 应用（React + TypeScript + Fastify + MySQL/MariaDB）
- **核心用户**：学校教务处、监考老师、学生、教室一体机
- **部署方式**：一键部署脚本 `install.sh v4.0`（Debian/Ubuntu 全自动，零技术门槛上线）

### 核心功能

| 功能模块 | 亮点 |
|---------|------|
| **四端协同 + JWT 互斥认证** | 公众前台 / 教室端大屏 / 教师端 / 管理员后台，四套独立 token，81 项越权测试全部 403 |
| **考试全流程管理** | 创建考试 → 分配教室（多对多）→ 分配监考教师 → 学生按班级关联 → 教室大屏倒计时 |
| **教室端监考模式** | 全屏超大字号倒计时（text-8xl）、5 分钟红色脉冲预警、进度条、Fullscreen API 一键全屏 |
| **AI 助手（Function Calling）** | 管理员用自然语言操作后台，27 个工具分 safe/dangerous/forbidden 三级，危险操作二次确认 |
| **一键部署 + 安装向导** | `install.sh` 全自动部署（DNS 校验 + SSL 四选项），浏览器向导 4 步初始化 |
| **安全加固体系** | JWT 强密钥校验、IP 黑名单、域名访问控制、XSS 双重过滤、速率限制、审计日志 |

## 二、Demo 创作思路

### 灵感来源——我自己就是学生

> "每次考试，最麻烦的不是复习，而是经常不知道具体时间和监考老师——那张贴在门口的通知纸太容易弄丢了，有时候老师也不会写黑板，全靠口头说，有的教室连个钟都没有，电脑上的时间小得坐在后排根本看不清。"
>
> —— 项目作者 落梦陳

于是我开始想：**教室里那块价值上万元的 86 吋一体机大屏，为什么不能直接显示考试信息？** 科目、时间、监考老师，把字放到最大，再加一个倒计时。这就是 KeKe ExamHub 的起点。

### 想解决的真实痛点

**痛点 1｜信息一致性灾难**

教务处老师在考试周需要逐个教室手写考试安排表，字迹因老师不同而千差万别。一所 36 间教室的学校，一次期中考试光手写通知就要跑遍整栋楼，往返耗时超过 2 小时。一旦考试时间调整，必须重新擦、重新写，信息出错率极高——曾有学校因手写通知时间写错导致全年级学生错过考试。

**痛点 2｜一体机资源巨额浪费**

教室里的交互式一体机（希沃、鸿合等品牌，65~86 英寸，单价过万），90% 时间只充当"高清投影仪"——上课投屏 PPT，下课挂着静态电子班牌。考试信息展示是一个高频刚需场景，却没有任何专用软件来承接，一体机的数字化价值被彻底搁置。

**痛点 3｜沟通链路脆弱**

考试安排依赖教师群→班级群→家长群的"传话游戏"。每经过一层转发，信息就有一次失真风险。最终学生和家长需要频繁私信确认，大量重复答疑消耗班主任和教务处精力。

**痛点 4｜无法全局统筹**

管理员用 Excel 排表，无法直观判断同一教学楼内是否存在教室冲突、同一监考老师是否被同时安排到两个考场、同一班级是否在同一时段被安排多门考试。

**痛点 5｜学生信息获取最困难**

学生作为考试信息的最终消费者，获取途径却最不可靠。一纸通知风吹即散，口头传达听过即忘，黑板板书擦完即无。没有人永久留存，没有人实时更新，没有人在意坐在后排的学生能不能看清楚。

### 为什么做这个方向

1. **真实需求**：这是我自己和身边同学真实遇到的痛点，不是凭空想象的概念
2. **社会价值大**：完全开源免费，让经费紧张的乡镇中小学也能拥有数字化考试管理能力
3. **落地可行性高**：Web 形态，无需安装 App，教室一体机打开浏览器就能用
4. **TRAE 最适合**：全栈项目，前后端 + 数据库 + 部署脚本 + AI 集成，正好发挥 TRAE 全流程开发能力

### 4.3 四端 JWT 互斥认证体系

这是项目最核心的安全设计。用 TRAE 实现了四套独立的认证中间件：

- `authMiddleware` — 管理员端（校验 role === "admin"）
- `classroomAuthMiddleware` — 教室端（校验 role === "classroom"）
- `studentAuthMiddleware` — 学生端（校验 role === "student"）
- `teacherAuthMiddleware` — 教师端（校验 role === "teacher"）

每个 token 携带 `role` 字段，中间件严格校验角色匹配。**81 项安全边界测试全部通过**：无 token → 401，跨角色访问 → 403。

额外的安全细节也都是和 TRAE 一起打磨的：
- 时序攻击防护：账号不存在时仍执行 bcrypt 比较均衡响应时间
- 速率限制：管理员登录 5 次/分钟，学生登录 10 次/分钟
- 弱密钥检测：JWT_SECRET 长度不足或在黑名单中时进入"仅安装模式"

### 4.5 AI 助手——Function Calling 三级安全体系

AI 助手是开发过程中用 TRAE 新增的亮点功能。管理员可以用自然语言操作后台：

- 「帮我查询下周有几场数学考试」
- 「创建一场英语期末考试，7月20日上午9点，在教学楼A101」
- 「把高三1班所有学生的信息导出来」

和 TRAE 一起设计了**三级安全执行体系**：
- **safe（13 个）**：纯查询类，自动执行（如查询考试、统计数据）
- **dangerous（14 个）**：写入类，前端弹窗二次确认才执行（如创建考试、导入学生）
- **forbidden**：删除类/高危操作，AI 直接拒绝并引导用户手动操作

技术实现上用 TRAE 解决了很多细节问题：SSE 流式响应、30 秒工具超时、4000 字符结果截断、上下文滑窗（30 条）、失败重试机制。

### 4.6 install.sh v4.0——工业级一键部署脚本

用 TRAE 写了 1280 行的一键部署脚本，从 v1 迭代到 v4.0：

- **错误处理**：`set -Eeuo pipefail` + `trap ERR` 全局错误捕获
- **日志落盘**：全程写入 `/var/log/examhub-install.log`，出问题可追溯
- **断点续传**：`--step <name>` 参数，失败后从指定步骤恢复
- **DNS 校验**：显式查询 8.8.8.8 / 1.1.1.1 / 114.114.114.114 / 223.5.5.5 公共 DNS，CDN/NAT 场景提供强制使用选项
- **SSL 四选项**：不配置 / 上传证书 / 检测已有 / 自动申请 Let's Encrypt
- **Nginx 动态配置**：根据场景（域名+SSL / 域名无SSL / 无域名）自动生成配置

最难忘的一个坑：GNU grep ERE 不支持 `\d`，DNS 解析校验一直失败，和 TRAE 一起排查了半天才找到原因，改成 `[0-9]` 就好了。

### 4.7 安全审计与 v1.2.0 发布——和 TRAE 一起查漏洞

发布前让 TRAE 对整个项目做了全面安全审计，发现了多个漏洞并逐一修复：

| 漏洞 | 严重程度 | 修复方式 |
|------|---------|---------|
| acme.ts 命令注入（execSync 拼接 shell） | 严重 | 改用 execFileSync 参数数组 |
| setup.ts 明文密码写入 .env | 严重 | 安装后移除 ADMIN_PASSWORD |
| acme.ts 路径遍历（domain 参数未校验） | 高 | 增加域名格式校验 |
| auth.ts IP 伪造（信任 X-Forwarded-For） | 高 | 统一使用 getClientIp |
| 前端公告 XSS（dangerouslySetInnerHTML） | 高 | 增加 DOMPurify 二次净化 |
| 分页接口 LIMIT 参数 500 错误 | 高 | pool.execute → pool.query |
| 注册码 Math.random 可预测 | 中 | 改用 crypto.randomInt |
| 密码仅长度要求，无复杂度 | 中 | 增加字母+数字复杂度校验 |

整个审计和修复过程完全在 TRAE 中完成，从发现问题、分析根因到写修复代码、重新测试，一条龙搞定。

### 4.8 开发踩坑记录

开发过程中遇到了不少坑，都是和 TRAE 一起逐一解决的：

1. **ESM 模块 `__dirname` 问题**：后端 `"type": "module"` 模式下 `__dirname` 不可用，用 `import.meta.url` 替代
2. **mysql2 预处理 LIMIT 参数 bug**：`pool.execute()` 配合 `LIMIT ? OFFSET ?` 报参数错误，改为 `pool.query()` 解决
3. **DNS 解析正则坑**：GNU grep ERE 不支持 `\d`，必须用 `[0-9]`
4. **pm2 spawn 崩溃**：pm2 未安装时 `spawn pm2 ENOENT` 会导致进程崩溃，添加 `.on("error")` 事件监听器
5. **安装向导建表不全**：最初只手动建了 9 张表，缺少 9 张关键业务表，改为加载完整 init.sql
6. **init.sql 的 USE 语句权限问题**：init.sql 含 `USE examhub` 导致用户自定义数据库名时报权限错误，移除后解决

## 五、价值与意义

### 效率提升（量化对比）

| 操作 | 传统方式 | KeKe ExamHub | 提升倍数 |
|------|---------|-------------|---------|
| 发布全年级考试安排 | 2 名老师、2 小时（手写 36 间教室通知） | 1 人、5 分钟（后台批量录入） | **24 倍** |
| 修改考试时间 | 跑楼重写通知 + 拍照重发群（约 30 分钟） | 后台编辑 → 全局实时刷新（约 10 秒） | **180 倍** |
| 学生查询考试信息 | 翻群聊/问同学/去公告栏（2~5 分钟） | 打开网页即看到全部安排（5 秒） | **24~60 倍** |
| 教室端信息更新 | 人工逐个教室切换画面或手写 | 自动刷新（30 秒间隔），零人工 | ∞ |
| 系统部署上线 | 找技术人员搭环境（半天到一天） | 一键脚本 + 浏览器向导（10 分钟） | 新能力 |

### 社会价值

**降低教育数字化转型门槛**

项目完全开源免费（MIT 协议）。对于经费紧张的乡镇中小学、偏远地区学校、民办教育机构而言，无需购买任何商业考试管理系统（市场年费通常 3000 元至 5 万元不等），即可拥有完整的数字化考试信息管理能力。

**推动一体机设备价值觉醒**

让全国数百万间教室里的交互式一体机从"高清投影仪"升级为"数字考试终端"，在不增加任何硬件投入的前提下将设备利用价值提升数倍。这一思路同样适用于电子班牌、校园信息发布屏等校园数字终端。

**安全可控的注册码体系**

教室端采用注册码注册 + 管理员审核的双重准入机制。管理员批量生成 `XXXX-XXXX` 格式注册码（去除易混淆字符 I/O/0/1），每码一次性使用。管理员对每一间接入系统的教室端拥有完全控制权：通过、驳回或删除。
